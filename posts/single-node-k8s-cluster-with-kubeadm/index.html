    WHY?!#

    Today I’m gonna write about setting up a tiny (single node) Kubernetes cluster using kubeadm inside a clean, generic Linux system. You won’t find anything about k3s, k0s, Talos nor RKE2 here. It’s far from Kubernetes The Hard Way approach in terms of sheer knowledge however it still provides a fair amount of learning opportunity without a premade script / dedicated OS. So answering the “why?!” question - education is a number one priority here.

    Prerequisites#

    It may surprise, but not a lot is actually needed to begin with Kubernetes. For such testing/learning purposes any spare hardware or a simple Virtual Machine will suffice.

    In my case a new VM was created on the Proxmox VE server I run and the following resources were provided:

    • 4 CPU
    • 6 GB of RAM
    • 60 GB of Storage
    • Network Interface with access to my local network and the internet Depending on available resources, the presented configuration can be scaled both ways however going extremally low will eventually cause issues.

    For the Linux distro, I decided to use Rocky Linux which is a fairly new, community driven RHEL-like distribution. I grew quite found of CentOS for years and was eager to finally try Rocky. Any Debian-based distro will also do the job but some commands will differ!

    Preparations#

    Kernel configuration#

    modules#

    Proper kernel modules are required for k8s and containers to run.

    sudo dnf install kernel-devel-$(uname -r)
sudo modprobe -a br_netfilter ip_vs ip_vs_rr ip_vs_wrr ip_vs_sh overlay

    To make the modules persist at reboots a configuration file has to be created.

    /etc/modules-load.d/kubernetes.conf

    br_netfilter
ip_vs
ip_vs_rr
ip_vs_wrr
ip_vs_sh
overlay

    sys.ctl#

    Kubernetes needs following settings to process network traffic and packets forwarding within the cluster. Another configuration file is necessary.

    /etc/sysctl.d/kubernetes.conf

    net.ipv4.ip_forward = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.bridge.bridge-nf-call-iptables = 1

    Config has to be applied with the following command.

    sysctl --system

    Swap#

    Kubernetes wasn’t designed to work with swap partition due to challenging resource management and interference with Quality of Service (QoS) guarantee. Even though it has recently became officially supported I wont be dealing with any of that for now.

    sudo swapoff -a
sudo sed -e '/swap/s/^/#/g' -i /etc/fstab

    Firewall#

    To allow internal and external traffic, the firewall needs to allow communication through specific ports

    sudo firewall-cmd --permanent --add-port={6443/tcp,2379/tcp,2380/tcp,10250/tcp,10259/tcp,10257/tcp,4789/udp}

    Because the goal of this blogpost is a highly unpractical scenario (single machine for both control plane and a worker node), additional ports can be opened to expose services via NodePort. Note that normally it shouldn’t be done on an any ControlPlane, ever.

    sudo firewall-cmd --permanent --add-port={30000-32767/tcp,30000-32767/udp}

    What are those ports for? Lemme explain!

    PortDescription
    2379-2380etcd server and client API
    4789VXLAN backend for flannel
    6443Kubernetes API server
    10250kubelet API
    10257kube-controller-manager
    10259kube-scheduler
    30000-32767NodePort Services
    After all that shenanigans, those changes have to be applied
    sudo firewall-cmd --reload

    Container runtime#

    A container runtime is needed for containerized apps to be run inside Pods. Containerd is an industry-standard in that matter and a considered a default with k8s.

    Containerd installation#

    First, Docker Community Edition repository has to be added to the package manager. After updating repository’s cache, containerd can be installed.

    sudo dnf config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
sudo dnf makecache
sudo dnf install containerd

    Containerd configuration#

    Before proceeding, Containerd has to be configured to use Systemd Cgroup. As a starting point I’m going to use the default config file which is generated by running

    sudo sh -c "containerd config default > /etc/containerd/config.toml";

    In the newly created file, SystemdCgroup entry has to be changed from false to true under `[plugins.“io.containerd.grpc.v1.cri”.containerd.runtimes.runc.options] key.

    /etc/containerd/config.toml

    [...]
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
            BinaryName = ""
            CriuImagePath = ""
            CriuPath = ""
            CriuWorkPath = ""
            IoGid = 0
            IoUid = 0
            NoNewKeyring = false
            NoPivotRoot = false
            Root = ""
            ShimCgroup = ""
            SystemdCgroup = false # <----  from false  to true!
[...]

    It can also be quickly achieved by running a sed command

    sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml

    Enabling containerd daemon#

    With the config.toml saved the containerd deamon can be enabled to start on boot.

    sudo systemctl enable --now containerd.service

    After system reboot, containerd.service is verified to be up and running

    sudo systemctl status containerd.service

    The actual k8s part#

    After all this tedious prep it’s finally time to do something with kube in its name. Three packages are required to setup, run and manage the cluster: kubelet,kubeadm and kubectl. It’s crucial for their versions to match and to be in full control when it comes to updates. I chose the newest version for the time, which is 1.33.

    Kubernetes packages installation#

    First, Kubernetes repository has to be added to the package manager.

    /etc/yum.repos.d/kubernetes.repo

    [kubernetes]
name=Kubernetes
baseurl=https://pkgs.k8s.io/core:/stable:/v1.33/rpm/
enabled=1
gpgcheck=1
gpgkey=https://pkgs.k8s.io/core:/stable:/v1.33/rpm/repodata/repomd.xml.key
exclude=kubelet kubeadm kubectl cri-tools kubernetes-cni

    The following commands are issued to begin package installation

    sudo dnf makecache
sudo yum install -y kubelet kubeadm kubectl --disableexcludes=kubernetes

    When the installation was done I enabled the kubelet service which is an optional step according to the documentation. This service will keep restarting until the actual cluster is up and running.

    sudo systemctl enable --now kubelet.service

    Initializing Kubernetes cluster with kubeadm#

    It is time, let there be cluster!

    sudo kubeadm init --pod-network-cidr=10.244.0.0/16 --skip-token-print

    The --pod-network-cidr=10.244.0.0/16 flag is necessary for the simplest networking plugin I’m going to use (for now). The --skip-token-print flag disables printing a token which is required for connecting additional nodes. I’ve put it there just to flex.

    Running the above command will initiate a Control Node creation and shouldn’t be problematic UNLESS any of preparation steps were omitted (which should be clear in the debug output) OR if someone insisted on running the node inside a Proxmox LXC (don’t ask how I know).

    kubeconfig file#

    In order to use kubectl from an user account, the configuration file has to be copied to that user’s home directory. It is encouraged because kubectl doesn’t like to be run as root and it may impose implications.

    sudo cp /etc/kubernetes/admin.conf $HOME/
sudo chown $(id -u):$(id -g) $HOME/admin.conf
export KUBECONFIG=$HOME/admin.conf

    Verification is in order!

    kubectl get nodes

NAME            STATUS      ROLES           AGE    VERSION
k8s-node-0      NotReady    control-plane   42s    v1.33.1

    Such an output gives two important information. The cluster was successfully initialized and the node is NotNoding… I mean NotReady. Waiting a bit more won’t resolve that issue. Trust me. So why is that?

    Networking plugin installation#

    Initializing a cluster doesn’t equip it with any networking capabilities. For Pods to communicate with each other a Networking Plugin is required. There are many to choose from and even though cilium seems to be a current go-to, this time I went with flannel - just to keep things simple and write another blogpost about replacing it eventually.

    kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml

    Above command will apply a manifest file from flannel’s repository, deploying flannel agent on the cluster. It’s preconfigured to use 10.244.0.0/16 cidr subnet and VXLAN as a backend - which explains opening the additional port a specifying that subnet during kubeadm init. After a short while kubectl get nodes should show the node being finally Ready

    Another command worth trying is kubectl get pods -A which will list all running Pods across available namespaces. They should all be running by now or at least getting there.

    From a Master to a Worker#

    Enabling a Control Plane node to work for itself is absolutely trivial and requires only removing the taint called :NoSchedule. It’s achieved by running the following command

    kubectl taint nodes <node-name> node-role.kubernetes.io/control-plane-

    The - at the end is what actually clears the taint from given node without changing its role

    Optionally, a worker label may be applied however it has no scheduling purpose, only organizational

    kubectl label node <node-name> node-role.kubernetes.io/worker=""

    As a bonus for my future self - this is how to reapply the:NoSchedule taint, duh.

    kubectl taint nodes <node-name> node-role.kubernetes.io/control-plane=:NoSchedule

    Testing, testing, 1-2-3#

    What’s better for testing purposes than good old NGNIX. The following manifest file creates a testnamespace, spawns a deployment called nginx with 3 replicas and exposes them through a NodePort service - also called ngnix.

    nginx-test.yaml

    ---
apiVersion: v1
kind: Namespace
metadata:
  name: test
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  namespace: test
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx
  namespace: test
spec:
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80
  type: NodePort

    Presented manifest can be applied by running:

    kubectl apply -f nginx-test.yaml

    To check if Pods were actually created and which port was assigned to the service

    kubectl get pods -n test
NAME                   READY   STATUS    RESTARTS   AGE
nginx-96b9d695-264zx   1/1     Running   0          15m
nginx-96b9d695-5xcns   1/1     Running   0          15m
nginx-96b9d695-6l8nx   1/1     Running   0          15m

kubectl get svc -n test
NAME    TYPE       CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
nginx   NodePort   10.104.29.57   <none>        80:30716/TCP   15m

    And finally, by accessing IP Address of the Node (not the CLUSTER-IP listed) with assigned port which in this case is 30716, the default nginx welcome message is displayed🎉
