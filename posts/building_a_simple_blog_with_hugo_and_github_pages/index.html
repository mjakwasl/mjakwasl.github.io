    Make yourself a blog, they said#

    Creating your own blog is a quite simple and pleasant way to document activities, share ideas and solutions to various problems as well as - especially in this case - an opportunity to learn something new.

    Why Hugo and GitHub Pages?#

    Despite the existence of many established platforms for bloggers (Medium, Blogger, Substack) and various CMS projects allowing to host your own, full-fledged website (Ghost, Strapi, TinaCMS, WordPressü§Æ) for the needs of this blog I decided to combine simplicity, comfort and elegance in the form of static pages fully hosted by GitHub Pages and built with Hugo.

    Hugo#

    Static site generator written in Go Lang. Fast, well known and enjoyed by many. It builds pages based on .md files (written in the Markdown language) and can be configured with .yaml, .json and .toml (by default). Hugo allows creating templates, full freedom of taxonomy and modular directory structure. What is particularly cool, it allows you to easily create translated versions of content. It’s FOSS btw.

    I got convinced to Hugo by an active community, nice free themes and suitable documentation. This static site generator can be combined with any headless CMS witch I might do in the distant future‚Ñ¢ but for the current needs I will just keep writing content in Obsidian and do version control strictly with GIT.

    GitHub Pages#

    Hosting service for static pages stored in GitHub repositories. Can be utilised to deploy a small personal website/blog (today’s goal) or pages dedicated to individual projects, e.g. API docs. GitHub Pages in its free plan has some limitations - it allows hosting only from your public repositories. Although in the case of this blog there is nothing preventing the entire source code to be publicly available (except for drafts, I guess), still I decided to get around this limitation by creating two separate repositories and had little fun with GitHub Actions.

    Requirements#

    • GitHub account
    • hugo package installed on your system or built from source
    • git installed and configured on the system (either with SSH key, plugin, client etc.)

    Depending on your OS the installation of required packages may differ. For most Linux distros hugo can be found inside common package managers, winget on Windows and homebrew on MacOS also know about Hugoüåû

    Creating GitHub repositories#

    Due to the mentioned free plan restrictions and my desire for source code to be hidden, 2 repositories will be necessary:

    • public one called <user>.github.io
    • private one with a chosen name, e.g. <your_blog>

    If making the source code public is not an issue or if you have a paid GitHub plan, one repository and the default Action code will suffice.

    Making a website with Hugo#

    Assuming that hugo and gitare already installed, begin by creating the website.

    Project initialization#

    hugo new site <your_blog> --format yaml

    It is worth to initialise the version control inside the created directory immediately.

    cd <your_blog>
git init

    Theme installation#

    Let’s start by choosing the preferred theme.

    There are several ways of theme installation but I’ll focus on the one recommended by the authors - using git submodule. The following command will download the PaperMod theme. Make sure to provide adequate URL and PATH if other theme was chosen. The theme will appear under /themes directory.

    git subemodule add --depth=1 https://github.com/adityatelange/hugo-ppermod.git themes/papermod

    Basic configuration#

    Edit the hugo.yamlto configure some basic settings.

    hugo.yaml

    baseUrl: https://examample.org/
languageCode: en-us
title: My New Hugo Site
theme: ["papermod"]
caches:
  images:
    dir: :cacheDir/images
    1. baseUrl- production URL of the website, in this case - https://<username>.github.io
    2. languageCode- language and country code, chose according to your needs
    3. title- website name (displayed by default in the header, footer and browser card)
    4. theme- Hugo needs you to specify which theme to use
    5. caches- since we will host the page on GitHub Pages, provided config is necessary Such a config is sufficient for now but feel free to check all possible parameters.

    First content and local test#

    Before we move on to the hosting part, we might want to test the blog locally. Let’s create the first entry on our blog

    hugo new content content/posts/first-entry.md

    This command will create a new .md file under /content/posts directory with a following content

    first-entry.md

    +++
title = 'First Entry'
date = 2025-06-04T16:20:00+01:00
draft = true
+++

    The default content on new entries is generated based on templates stored in /archetypes directory. Beside title and date of creation, new post receives an enabled draft parameter, witch is preventing publication of unfinished content.

    Let’s add a few lines of text and check what happens.

    first-entry.md

    +++
title = 'First Entry'
date = 2025-06-04T16:20:00+01:00
draft = true
+++
## Hello world!

It's my **first** entry on this *blog*.

    Save the file and launch Hugo server with an additional flag, which will include drafts on build.

    hugo server -buildDrafts
OR
hugo server -D

    After entering the IP displayed in the console, you should see the “First Entry” post. As long as the development server is running, all changes will appear on the blog in real time. Finally, you can change the draft parameter to false.

    When Hugo server was first started, a new directory called /public appeared. It contains the static version of your website which we gonna push later to the separate, publicly available repository.

    Blog publication on GitHub Pages#

    In the following section I will present how to ommit the limitation of the free Pages plan which forces you to expose the source code(I mean drafts…) publicly. For this I used 2 separate repositories and modified the official Hugo on GH Pages deployment Action.

    .gitignore, first commit and push#

    If you haven’t done it already, it’s finally time to push your source code to the private repository. Before that, it is worth crating 2 files in the root directory. The most important is .gitignore. I suggest the following content:

    .gitignore

    /public
/resources
    • /public is a directory in witch static web build is placed into. Versioning is not necessary here, because GitHub Actions gonna take care of building and the public repo will keep track of the versions.
    • /resources stores cache files of processed images and in this case I also consider them useless

    Optionally, a README.md file can be created, describing the project.

    README.md

    ## Private blog repo
Made with Hugo
### Local development
To run a local development server use: `hugo server -D`

    Go ahead and push your code to the remote repository.

    git add .
git commit -m 'initial commit'
git remote add origin https://github.com/<user>/<repository>.git
git branch -M main
git push -u origin main

    Replace <user> and <repository> with the appropriate names. An error during a push might be caused by GIT not being properly configured, provided URL is wrong or the repository on the origin side has a conflict with the local one. If everything went right, your private repo should contain the source code.

    Deploying with GitHub Actions#

    In the root directory of your website create a .github folder and a Hugo.yml file under it.

    .github/hugo.yml

    name: Build with Hugo and push to a public repo
on:
  push:
    branches: ["master"]
  workflow_dispatch: null
defaults:
  run:
    shell: bash
jobs:
  build:
    runs-on: ubuntu-latest
    env:
      HUGO_VERSION: 0.147.7
    steps:
      - name: Install Hugo CLI
        run: >
          wget -O ${{ runner.temp }}/hugo.deb https://github.com/gohugoio/hugo/releases/download/v${HUGO_VERSION}/hugo_extended_${HUGO_VERSION}_linux-amd64.deb && sudo dpkg -i ${{ runner.temp }}/hugo.deb
      - name: Install Dart Sass
        run: sudo snap install dart-sass
      - name: Checkout
        uses: actions/checkout@v4
        with:
          submodules: recursive
      - name: Install Node.js dependencies
        run: '[[ -f package-lock.json || -f npm-shrinkwrap.json ]] && npm ci || true'
      - name: Build with Hugo
        env:
          HUGO_CACHEDIR: '${{ runner.temp }}/hugo_cache'
          HUGO_ENVIRONMENT: production
        run: |
          hugo \
            --minify \
            --baseURL "https://${{ github.actor }}.github.io"
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public
  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Download built site artifact
        uses: actions/download-artifact@v4
        with:
          name: github-pages
          path: ./tmp-artifact
      - name: Exctract artifact
        run: |
          mkdir ./public
          tar -xf ./tmp-artifact/artifact.tar -C ./public
      - name: Checkout target repo
        uses: actions/checkout@v4
        with:
          repository: '${{ github.actor }}/${{ github.actor }}.github.io'
          token: '${{ secrets.TARGET_REPO_PAT }}'
          path: ./public-repo
      - name: 'Sync directories'
        run: rsync -av --delete --exclude .git --exclude README.md  ./public/ ./public-repo/
      - name: Configure git
        run: |
          git config --global user.name "github-actions"
          git config --global user.email "github-actions@github.com"
      - name: 'Commit and push to public repo'
        run: |
          cd public-repo
          git add .
          git commit -m "Deploy from Hugo site - ${{ github.sha }}" || echo "No
          changes to commit"
          git push

    Save that file and proceed by creating a secret called TARGET_REPO_PAT within your private repository settings. The value of that secret should be classic Token created in your account’s developer settings . Select only “repository” permissions and an adequate expiration time. Make sure that your public repository is called <your_username>.github.io.

    Add, commit and push the changes. Go to the Actions tab in your private repo and observe how the site is building. If everything went successfully, the public repo should receive your website build, and your blog should be accessible under https://<your_username>.github.io.

    In case of errors, it is worth checking whether the Hugo version in Actions matches the one used for local development, token validity, its permissions and the name of your public repository.
