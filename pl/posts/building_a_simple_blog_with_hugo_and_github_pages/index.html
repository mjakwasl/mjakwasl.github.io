    Zrób bloga, mówili#

    Tworzenie własnego bloga to dzisiaj całkiem prosty i przyjemny sposób na dokumentowanie swoich poczynań, dzielenie się pomysłami i rozwiązaniami rozmaitych problemów a także - jak w tym przypadku - okazja do nauki czegoś nowego.

    Dlaczego Hugo i GitHub Pages?#

    Pomimo istnienia wielu zintegrowanych platform dla blogerów (Medium, Blogger, Substack) oraz rozmaitych CMS pozwalających hostować własne, rozbudowane strony (Ghost, Strapi, tinaCMS, Wordpress🤮) dla potrzeb tego bloga zdecydowałem się na kompromis prostoty, wygody i elegancji w postaci statycznych stron budowanych przez Hugo, hostowanych w pełni przez GitHub Pages.

    Hugo#

    Generator stron statycznych napisany w Go. Szybki, znany i lubiany. Buduje strony na podstawie plików .md czyli napisanych w języku znacznikowym (markup), a jego konfiguracje możemy pisać w plikach .yaml, .jsonlub domyślnie .toml - do wyboru do koloru. Pozwala na tworzenie templatek, pełną dowolnośc taksonomii oraz struktury plików, konfikuracja jest modularna a co szczególnie fajne, pozwala w prosty sposób tworzyć dodatkowe wersje językowe naszych wpisów i stron. Jest to oczywiście w pełni darmowe oprogramowanie z otwartym kodem źródłowym (FOSS).

    Do Hugo przekonała mnie przede wszystkim aktywna społeczność, ładne motywy i przyjemna dokumentacja. Generator ten można połączyć z dowolnym headless CMSem co może w przyszłości™ uczynię ale na obecne potrzeby zostanę przy pisaniu treści w Obsidianie i zwykłym wersjonowaniu przez git’a.

    GitHub Pages#

    Serwis hostujący statyczne strony przechowywane w repozytoriach GitHub. Możemy z jego pomocą udostępnić swoją małą personalną stronkę/blog (na czym skupimy się w tym wpisie) lub strony dedykowane poszczególnym projektom, np. dokumentację API. GitHub Pages w darmowej wersji konta posiada pewne ograniczenie - pozwala hostować jedynie z publicznych repozytoriów. Choć w przypadku tego bloga nic nie stoi na przeszkodzie upublicznieniu całego kodu źródłowego (może poza ujawnieniem wersji roboczych wpisów) to i tak postanowiłem obejść to ograniczenie dzięki dwóm osobnym repozytoriom oraz małej zabawy w GitHub Actions.

    Wymagania wstępne#

    • pakiet hugo zainstalowany dowolną metodą LUB zbudowany z kodu źródłowego
    • konto w serwisie GitHub oraz zainstalowany git na kompie
    • opcjonalnie skonfigurowany klucz SSH do autoryzacji z prywatnym repozytorium

    W zależności od systemu operacyjnego instalacja wymaganych pakietów może się odrobinę różnić. W przypadku większości dystrybucji Linux hugo znajduje się w domyślnych repozytoriach menadźerów pakietów, winget na Windows oraz homebrew na MacOS również znają się z Hugo 🌞

    Tworzenie repozytoriów na GitHub#

    Ze względu na wspomniane ograniczenia darmowego konta i chęć nieupubliczniania pełnego kodu niezbędne będą 2 repozytoria:

    • publiczne o nazwie <użytkownik>.github.io
    • prywatne o downolnej nazwie np. <nazwa_strony>

    Jeżeli publikacja całego kodu to nie problem lub jeśli posiadasz płatne konto w serwisie GitHub to jedno repozytorium jest całkowicie wystarczające.

    Tworzenie strony z Hugo#

    Przy założeniu że hugo oraz git są już zainstalowane, możemy przystąpić do tworzenia strony.

    Inicjalizowanie projektu#

    hugo new site <nazwa_strony> --format yaml

    To polecenie stworzy nowy folder <nazwa_strony> zawierający podstawową strukturę projektu Hugo. Wyglądać to będzie nastepująco:

    <nazwa-strony>/
├── archetypes/
│   └── default.md
├── assets/
├── content/
├── data/
├── i18n/
├── layouts/
├── static/
├── themes/
└── hugo.yaml

    Paramater --format yaml przekazałem w celu zmiany domyślnego rozszerzenia plików konfiguracyjnych z .toml na bardziej czytelny i wygodny .yaml.

    Warto od razu zinicjalizować kontrolę wersji wewnątrz stworzonego katalogu

    cd <nazwa_strony>
git init

    Instalacja motywu#

    Zacznijmy od wybrania preferowanego motywu. Obszerną listę znajdziemy chociażby tutaj. Sposobów na instalację jest kilka. Możemy skorzystać z dedykowanego modułu Hugo (choć wymaga to instalacji go lang), wypakować pobrane archiwum zip do folderu themes lub sklonować tam repozytorium motywu. Ja wybrałem czwartą, rekomendowaną przez autorów opcję, czyli git submodule. Poniższa komenda prezentuje instalacje na przykładzie motywu PaperMod.

    git submodule add --depth=1 https://github.com/adityatelange/hugo-PaperMod.git themes/PaperMod

    W katalogu /themes pojawi się wybrany przez nas motyw. Musimy dać znać Hugo, żeby z niego korzystał. Odpowiada za to wpis w pliku hugo.yaml

    theme: ["PaperMod"]

    Podstawowa konfiguracja#

    Pozostając w pliku hugo.yaml możemy skonfigurować kilka podstawowych ustawień.

    hugo.yaml

    baseURL: https://example.org/
languageCode: en-us
title: My New Hugo Site
theme: ["PaperMod"]
caches:
  images:
    dir: :cacheDir/images
    1. baseUrl - przyszły adres produkcyjny naszej strony, jeżeli nie posiadasz własnej domeny i korzystać będziesz z GitHub Pages, wpisz https://<użytkownik>.githubpages.io
    2. languageCode - domyślny język i region, jeżeli blog będzie głównie w języku polskim to ustaw pl-pl
    3. title - nazwa strony/bloga, wyświetlana m.in. w nagłówku oraz w karcie przeglądarki
    4. theme - wybrany przez nas motyw
    5. caches - dodatkowe ustawienie wymagane z racji, że strona hostowana będzie na GitHub pages

    Taka konfiguracja na poczatek jest w zupełności wystarczająca. Tutaj znajdziesz pełną listę możliwych ustawień

    Pierwszy wpis#

    Zanim przejdziemy do udostępnienia strony w GitHub Pages, warto ją przetestować lokalnie. Przygotujmy na tą potrzebę pierwszy wpis.

    hugo new content content/posts/pierwszy-wpis.md

    Polecenie to utworzy nowy plik .md w katalogu /content/posts o następującej zawartości

    pierwszy-wpis.md

    +++
title = 'Pierwszy Wpis'
date = 2025-06-04T16:20:00+01:00
draft = true
+++

    Domyślna zawartość nowych wpisów generowana jest z templatek znajdujących się w katalogu /archetypes. Na podstawie wbudowanej templatki, oprócz tytułu i daty stworzenia, nasz nowy post otrzymuje również aktywowany parametr draft, zapobiegający publikacji nieukończoncyh treści. Dodajmy kilka linijek tekstu i sprawdźmy co się stanie.

    pierwszy-wpis.md

    +++
title = 'Pierwszy Wpis'
date = 2025-06-04T16:20:00+01:00
draft = true
+++
## Witaj świecie!

To mój **pierwszy** wpis na tym *blogu*.

    Plik zapisujemy i uruchamiamy Hugo poleceniem z dodatkową flagą, która uwzględni przy budowaniu wpisy oznaczone jako draft.

    hugo server --buildDrafts
ALBO
hugo server -D

    Po wejściu w wyświetlony przez konsolę adres zobaczymy stworzony przez nas post. Tak długo jak uruchomiony jest serwer developerski wszelkie zmiany będą pojawiać się na stronie w czasie rzeczywistym. Na zakończenie możesz zmienić paramter draft na false. Zaobserwować też można nowy katalog w struktorze naszego bloga o nazwie /public. To tutaj znajduje się zbudowana przez Hugo statyczna wersja naszej strony i to właśnie zawartość tego katalogu wypychać będziemy do publicznego repozytorium.

    Publikacja bloga przez GitHub Pages#

    W dalszej części przedstawię jak rozwiązałem problem darmowego planu Pages czyli publikację stron wyłacznie z publicznych repozytoriów przy jendoczesnej chęci zachowania kodu źródłowego prywatnie. Wykorzystałem do tego 2 osobne repo oraz zmodyfikowany oficialny deployment Hugo przez GitHub Actions.

    .gitignore, pierwszy commit i push#

    Jeżeli jeszcze tego nie uczyniłeś to najwyższy czas wysłać kod do repozytorium. Zanim to zrobimy warto dodać kilka istotnych plików w głównym katalogu. Najważniejszy to .gitignore, który wskazuje co pominąć podczas wersjonowania. Proponuję następującą zawartość:

    .gitignore

    /public
/resources
    • /public to katalog do którego trafia zbudowana przez Hugo strona, nie potrzebujemy jej wersjonować w tym repozytorium, ponieważ budowanie strony odpbywać się będzie w GitHub Actions a zawartość trafiać będzie do drugiego, publicznego repozytorium.
    • /resources przechowuje m.in. cache przeprocesowanych obrazów i w tym przypadku również uważam je za zbędne.

    Opcjonalnie możemy również dodać plik README.md z opisem naszego projektu.

    README.md

    ## Prywatne repozytorium bloga
Zbudowane przy pomocy Hugo
### Lokalny development
Lokalny server developerski uruchomisz komendą `hugo server -D`

    Tak przygotowany katalog wypychamy na serwery.

    git add .
git commit -m 'Pierwszy commit'
git remote add origin https://github.com/<użytkownik>/<repozytorium>.git
git branch -M main
git push -u origin main

    Zamień <użytkownik> i <repozytorium odpowiednimi nazwami. Jeżeli podczas push’a otrzymasz błąd to najprawdopodbniej git nie został odpowiednio skonfigurowany, podane dane się nie zgadzają lub w repozytorium już coś się znajduje. Jeżeli wszystko przebiegło pomyślnie to w repozytorium na GitHub powinien znajdować się kod.

    Tworzenie deploymentu w GitHub Actions#

    Ten krok można wykonać zarówno z poziomu interfejsu GitHub’a jak i z poziomu kodu w repozytorium - przedstawię tutaj tą drugą metodę. W głównym katalogu naszej strony tworzymy folder .github a w nim plik hugo.yml

    .github/hugo.yml

    name: Zbuduj stonę z Hugo i wypchnij do publicznego repozytorium
on:    #Automatyzacja urcuhamiać się będzie przy każdej zmianie w głownej gałęzi
  push:
    branches: ["master"]
  workflow_dispatch: # Oraz ręcznie :>
defaults:
  run:
    shell: bash
jobs:
  build:
    runs-on: ubuntu-latest
    env:
      HUGO_VERSION: 0.147.7  #Najnowsza wersja na  czas tworzenia tego wpisu
    steps:
      - name: Install Hugo CLI
        run: |
          wget -O ${{ runner.temp }}/hugo.deb https://github.com/gohugoio/hugo/releases/download/v${HUGO_VERSION}/hugo_extended_${HUGO_VERSION}_linux-amd64.deb && sudo dpkg -i ${{ runner.temp }}/hugo.deb
      - name: Install Dart Sass
        run: sudo snap install dart-sass
      - name: Checkout
        uses: actions/checkout@v4
        with:
          submodules: recursive
      - name: Install Node.js dependencies
        run: "[[ -f package-lock.json || -f npm-shrinkwrap.json ]] && npm ci || true"
      - name: Build with Hugo
        env:
          HUGO_CACHEDIR: ${{ runner.temp }}/hugo_cache
          HUGO_ENVIRONMENT: production
        run: |
          hugo \
            --minify \
            --baseURL "https://${{ github.actor }}.github.io"
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public
  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Download built site artifact
        uses: actions/download-artifact@v4
        with:
          name: github-pages
          path: ./tmp-artifact
      - name: Exctract artifact
        run: |
          mkdir ./public
          tar -xf ./tmp-artifact/artifact.tar -C ./public
      - name: Checkout target repo
        uses: actions/checkout@v4
        with:
          repository: '${{ github.actor }}/${{ github.actor }}.github.io'
          token: '${{ secrets.TARGET_REPO_PAT }}'
          path: ./public-repo
      - name: 'Sync directories'
        run: rsync -av --delete --exclude .git --exclude README.md  ./public/ ./public-repo/
      - name: Configure git
        run: |
          git config --global user.name "github-actions"
          git config --global user.email "github-actions@github.com"
      - name: 'Commit and push to public repo'
        run: |
          cd public-repo
          git add .
          git commit -m "Deploy from Hugo site - ${{ github.sha }}" || echo "No
          changes to commit"
          git push

    Plik zapisujemy i przechodzimy do stworzenia sekretu o nazwie TARGET_REPO_PAT w ustawieniach źródłowego (prywatnego) repozytorium. Wartością będzie token, który generuje się w ustawieniach developerskich konta . Typ tokenu wybieramy jako klasyczny oraz przydzielamy uprwanienia wyłacznie do repozytoriów. Skrypt zakłada, że na naszym koncie istnieje już publiczne repozytorium o nazwie <użytkownik>.github.io.

    Po upewnieniu się, że wszystko się zgadza robimy commit i push a następnie obserwujemy w panleu Actions jak nasza strona się buduje. Jeśli wszystko przebiegło pomyślnie to w publicznym repozytorium znajdować się będzie zbudowana strona, która z automatu dostępna jest pod adresem https://<użytkownik>.github.io. W przypadku błędów warto sprawdzić, czy wersja, którą budujemy stronę w Actions pokrywa się z tą, na której developujhemy stronę lokalnie, zweryfikować poprawnośc tokenu i jego uprawnienia oraz nazwę publicznego repozytorium.
